OWASP

  • Open Web Application Security Project (“Oh-wasp”)
  • Clearinghouse for good security practices
  • Also some good security information
  • Check out the juice shop application for instance!

Broken object level authorization

  • Can I get to (authorization) stuff (objects) that I shouldn’t be able to (broken)
  • Brute force is often useful here
  • Use the principle of least privilege on object authorization
  • Issue area: Application design and implementation

Broken user authentication

  • Can I impersonate (authentication) someone (user) I’m not (broken)?
  • Defensible with good password practice and hygiene
  • Principle of least privilege still applies
  • Issue area: Application design, platform hosting

Excessive data exposure

  • Returns too much (excessive) data
    • Too many records?
    • Too much about a record
  • Not really an attack, more of an accident
  • Think of SELECT * vs SELECT [fields]
  • Issue area: Application design and implementation

Lack of Resource and Rate Limiting

  • Denial of availability
  • Throttle requests handled
  • Tension between availability and vulnerability
  • Load testing usually covers some of this issue
  • Issue area: Platform hosting, sometimes application design and implementation

Broken function level authorization

  • Similar to broken object level authorization
  • Are you granted authorization to API functions that you shouldn’t be?
  • Combine with Lack of resource / DDoS type attacks
  • Use role-based access to reinforce limiting authorization
  • Issue area: Application design and implementation

Mass assignment

  • Data is converted into objects in the API
  • The attacker might add more data/fields which would then override important information
  • Issue area: Application design and implementation

Security misconfiguration

  • Something somewhere has a hole; very broad!
  • Often comes up between prototyping and production
    • Security is a pain when you’re rapidly developing
    • Security is complicated to stay on top of
  • Issue area: Platform hosting, some application design

Injection

  • An attacker’s attempt to send data to an application in a way that will change the meaning of commands being sent to an interpreter
  • Issue area: Application implementation, some design

Improper Assets Management

  • Bad environment and version management
    • Bad environments: Two environments share resources or configuration
    • Bad versions: Two or more versions share vulnerabilities
  • Issue area: Platform hosting, application design

Insufficient logging and monitoring

  • Risk because you’re not getting good information from your application
  • Not a “traditional” risk because it’s not directly exploitable
  • But a huge problem if you can’t detect changes (bad ones!) in your application
  • Issue area: Platform hosting, application design, implementation
